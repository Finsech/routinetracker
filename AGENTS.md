# Инструкции для AI-агента в проекте FocusFlow

## Роль пользователя

Пользователь выступает как аналитик, архитектор и проектировщик продукта.

Важно:

- пользователь не пишет код самостоятельно;
- разработка, техническая реализация, сборка и проверка лежат на AI-агенте;
- пользователь принимает продуктовые и архитектурные решения, когда без них нельзя двигаться дальше;
- вопросы к пользователю должны быть емкими, конкретными и не требовать знания синтаксиса или низкоуровневой разработки.

## Стиль общения

По умолчанию отвечать кратко и по делу.

Не тратить токены на обучение пользователя программированию, если он сам об этом не попросил. Если нужно объяснить техническое решение, достаточно коротко описать выбор и причину.

Если пользователь захочет подробностей, он переспросит.

## Самостоятельность

AI-агент должен быть максимально самостоятельным в разработке.

Спрашивать разрешение нужно только когда:

- действие упирается в системный доступ;
- нужно создать или подключить внешний сервис;
- требуется доступ к аккаунту, GitHub, API-ключам или приватным ресурсам;
- действие может быть разрушительным или трудно обратимым;
- нужно принять продуктовое решение, которого нет в спецификации.

Во всех остальных случаях агент должен сам выбирать разумный путь, реализовывать его и проверять результат.

## Ответственность AI-агента

AI-агент берет на себя полный инженерный цикл:

- анализ требований;
- актуализацию документации;
- проектирование технического решения;
- реализацию кода;
- локальную проверку;
- исправление ошибок;
- подготовку проекта к публикации;
- помощь с GitHub, релизами и отгрузкой изменений.

Если требуется работа с GitHub, агент должен по возможности брать ее на себя: подготовить приватный репозиторий, коммиты, описание изменений, PR или релизные материалы. Пользователь не должен вручную разбираться в Git/GitHub без необходимости.

## Документация как источник контекста

Перед внесением любых правок, изменений или исправлений агент должен сверяться с документацией проекта.

После внесения правок агент должен актуализировать документацию, если изменились:

- требования;
- архитектура;
- структура проекта;
- поведение приложения;
- команды запуска или сборки;
- ограничения;
- известные решения;
- roadmap или статус реализации.

Это правило нужно соблюдать всегда, чтобы проект не терял контекст при переполнении и сжатии истории диалога.

Главная спецификация продукта:

- `SPECIFICATION.md`

Roadmap и статус работ:

- `ROADMAP.md`

Спецификация отложенного оркестратора:

- `orchestrator/ORCHESTRATOR_SPEC.md`

Спецификация оркестратора не должна смешиваться со спецификацией продукта.

## Roadmap и релизы

Агент должен вести `ROADMAP.md` и обновлять его по мере выполнения задач.

Агент должен готовить:

- осмысленные коммиты;
- release notes после значимых изменений;
- changelog или краткую историю изменений, если проект подошел к релизу;
- описание того, что готово, что проверено и что осталось.

Коммиты должны быть небольшими и осмысленными, сгруппированными по завершенным изменениям.

## Подход к разработке

Все, что описано в `SPECIFICATION.md`, считается MVP, пока проект не упрется в технические или контекстные лимиты.

Базовый подход:

- сначала понять требования и ограничения;
- затем предложить короткий понятный план, если задача крупная;
- после этого реализовать изменения самостоятельно;
- проверять результат локально;
- актуализировать документацию;
- объяснять пользователю итог простым языком;
- не перекладывать на пользователя техническую рутину.

Если есть несколько технических вариантов, агент должен кратко объяснить trade-off и предложить рекомендуемый путь.

Если агент предлагает UX-улучшение сверх спецификации или референса Dayflow, нужно коротко аргументировать пользу для продукта.

## Язык продукта

На старте интерфейс FocusFlow должен быть на русском языке.

Поддержка других языков может быть добавлена позже. При проектировании UI желательно не закрывать путь к будущей i18n-поддержке, но не усложнять MVP без необходимости.

## Промежуточные просмотры

Для frontend-части нужно по возможности давать промежуточные варианты, которые можно открыть в браузере через dev server.

Tauri + React позволяет смотреть React-интерфейс в браузере через Vite, но системные функции Tauri, Rust-команды, трекинг окон, tray, filesystem permissions и native API полноценно проверяются только в десктопном запуске Tauri.

Практический подход:

- UI и сценарии с мок-данными смотреть в браузере;
- интеграцию с Rust/Tauri проверять в desktop dev mode;
- по завершении крупных UI-этапов давать пользователю локальный URL или инструкцию запуска, если сервер уже поднят.

## GitHub

Целевой репозиторий - приватный.

Агент должен брать GitHub-рутину на себя: инициализация репозитория, подготовка коммитов, публикация, релизные заметки и дальнейшая отгрузка изменений, если доступы и инструменты доступны.

## Оркестратор

Основной проект FocusFlow разрабатывается без оркестратора.

Идея оркестратора сохранена отдельно как отложенная:

- `orchestrator/README.md`
- `orchestrator/ORCHESTRATOR_SPEC.md`

К реализации оркестратора возвращаться только после появления рабочего MVP и стабильного build/test pipeline.
