# FocusFlow Specification

## 1. Цель продукта

FocusFlow — локальный desktop-трекер времени, который автоматически собирает цифровую активность пользователя и превращает шумный поток окон, вкладок и коротких переключений в понятную картину дня:

- **потоки** верхнего уровня;
- **стримы** внутри потоков;
- обзор дня и недели;
- локальную LLM-сводку без обязательной отправки данных в облако.

Продукт ориентирован на анализ реальной работы, а не просто на подсчет времени в приложениях.

## 2. Принципы MVP

1. **Local-first**
   - трек хранится локально;
   - база локальная;
   - LLM по умолчанию локальная через Ollama.

2. **Минимум ручной рутины**
   - трекинг работает сам;
   - приложение живет в трее;
   - пользователь не должен вручную запускать запись каждой сессии.

3. **Из шума в смысл**
   - FocusFlow должен уметь показать не только приложение, но и более понятный рабочий контекст:
     - поток;
     - стрим;
     - проект.

4. **Русский интерфейс на старте**
   - UI продукта в MVP — русский;
   - архитектура не должна ломать будущую i18n-поддержку.

## 3. Платформы

### Windows

Основная поддерживаемая платформа MVP:

- нативный трекинг окон;
- системный трей;
- автозапуск;
- packaged builds (`exe`, `msi`, `nsis`).

### macOS

Поддержка в стадии ранней дистрибуции:

- build собирается через GitHub Actions;
- распространяется unsigned `.dmg`;
- подходит для тестового запуска и open-source distribution;
- polished path потребует signing/notarization позже.

## 4. Основные экраны

### Сегодня

Показывает:

- таймлайн дня;
- агрегированные часовые полосы активности;
- подробности выбранного часа;
- блок `Твой день`;
- блок `Потоки и стримы`;
- ручной запуск LLM-группировки.

### Анализ дня

Показывает:

- обзор по выбранному дню;
- потоки дня;
- `Картина дня`;
- heatmap активности;
- выбор даты через datepicker.

### Неделя

Показывает:

- ритм недели;
- вертикальные day-bars;
- weekly summary;
- `Контексты недели` или `Контексты дня`;
- годовую heatmap активности.

### Настройки

Содержит:

- язык;
- тему;
- автозапуск;
- Ollama URL и модель;
- экспорт;
- stoplist;
- служебные настройки, которые нужны для MVP.

## 5. Трекинг

FocusFlow собирает:

- активное окно;
- заголовок окна;
- процесс / приложение;
- браузерный URL и title через локальный browser bridge;
- idle-интервалы.

В логике трекинга уже предусмотрены:

- корректная работа после `sleep / wake`;
- работа в трее;
- single-instance режим;
- повторное открытие уже запущенного экземпляра вместо второго процесса.

## 6. Потоки и стримы

### Потоки верхнего уровня

В текущем MVP используются:

- `Работа`
- `Обучение`
- `Общение`
- `Прочее`
- служебные/raw слои для внутреннего представления

### Стримы

Стрим — это более конкретный смысловой блок внутри потока:

- проект;
- набор связанных задач;
- коммуникация в определенном канале / мессенджере;
- учебный блок;
- прочий рабочий контекст.

## 7. LLM-пайплайн

### Текущий подход

Для LLM-саммаризации используется локальная Ollama.

Пайплайн:

1. сбор сырых активностей за день;
2. rule-based предклассификация;
3. chunking длинного дня;
4. локальная саммаризация чанков;
5. merge-pass;
6. пост-обработка и нормализация потоков/стримов.

### Что вынесено из дорогого LLM-пути

- явные коммуникации;
- idle;
- часть очевидного не-проектного хвоста;
- ряд простых контекстов, где LLM не нужна для базовой классификации.

### Что LLM должна решать лучше всего

- определение проектов внутри `Работы`;
- склейка похожих рабочих контекстов между браузером, IDE и смежными инструментами;
- нормализация итоговой структуры дня.

### Технические подсказки

В payload используются:

- `hint_flow`
- `project_hint`
- `project_key`

Это нужно, чтобы локальная модель лучше удерживала проектный контекст на длинных днях.

## 8. Коммуникации и рабочие кандидаты

### Коммуникации

Встроенный словарь включает как минимум:

- Telegram
- Slack
- WhatsApp
- Discord
- MAX
- Yandex Messenger
- web-версии этих инструментов

### Рабочие кандидаты

В rule-based слое отдельными рабочими кандидатами считаются:

- браузеры;
- IDE;
- редакторы кода;
- дизайн-инструменты;
- рабочие календари;
- инструменты созвонов;
- ряд типовых рабочих доменов.

## 9. Экспорт

Поддерживаются:

- `JSON` — полный экспорт/бэкап;
- `CSV` — плоский экспорт для табличного анализа.

## 10. Настройки и ограничения

### Что уже есть

- локальный Ollama provider;
- URL модели и имя модели;
- автозапуск Windows;
- stoplist;
- export options;
- пометки `Перевод в работе` / `Тема в работе` для незавершенных слоев.

### Что пока не реализовано

- внешний LLM provider по токену;
- полноценная i18n;
- полноценная theme system;
- polished signed macOS distribution.

## 11. Архитектура

### Desktop shell

- Tauri
- Rust backend

### Frontend

- React
- TypeScript
- Tailwind CSS
- UI copy и пользовательские ошибки постепенно выносятся в отдельные слои

### Storage

- SQLite

### Local AI

- Ollama
- prompt strings для LLM-пайплайна вынесены отдельно от продуктового UI-слоя

1. стабильность LLM на длинных днях;
2. качество project detection внутри `Работы`;
3. обкатка автосаммари;
4. post-release багфиксы без раздувания scope;
5. дальнейшая санитарная чистка кодовой базы и документации.
6. новые фичи и пост-релизные улучшения уже строятся поверх очищенного слоя строк и идентификаторов.
