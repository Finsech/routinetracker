FocusFlow - десктопное приложение для Windows 11, которое автоматически отслеживает экранное время, визуализирует активность и использует LLM для группировки действий по проектам и категориям.

## 1. Цель проекта

Создать локальное приложение для анализа цифровой активности пользователя. Приложение должно работать в фоне, фиксировать активные окна и сайты, определять периоды простоя, показывать понятную статистику и помогать пользователю видеть, на какие проекты и типы деятельности уходит время.

## 2. Ключевые возможности

- Автоматический трекинг активного окна и активности браузера.
- Определение простоя по отсутствию активности мыши и клавиатуры.
- Визуализация данных в стиле GitHub heatmap и дневных/недельных таймлайнов.
- Иерархическая LLM-саммаризация: активности -> стримы -> потоки.
- Фоновая работа без постоянного участия пользователя.
- Локальное хранение данных в SQLite.
- Приватный режим через локальную LLM, например Ollama.
- Повышенная стабильность через watchdog-процесс и мягкую периодическую перезагрузку.

## 3. Термины

- **Активность** - отдельный зафиксированный интервал работы в приложении или на сайте.
- **Стрим** - конкретный проект или задача, собранная LLM из сырых логов активности. Примеры: `Разработка FocusFlow`, `Анализ конкурентов`.
- **Поток** - высокоуровневая категория для одного или нескольких стримов. Примеры: `Работа`, `Обучение`, `Общение`, `Развлечения`, `Рутина`.
- **Простой** - период без активности мыши или клавиатуры дольше заданного порога.

- Экран настроек поддерживает экспорт данных в JSON. Выгрузка содержит `activity_log`, `idle_log`, `settings`, `stoplist`, сохраненные `llm_summary`, дату экспорта и версию схемы.

## 6. Архитектура

### 6.1. Бэкенд

Бэкенд реализуется на Rust внутри Tauri и отвечает за системную логику приложения.

Основные обязанности:

- Отслеживание активного окна через системные хуки ОС.
- Управление локальной базой данных SQLite.
- Запуск локального HTTP bridge для связи с расширением браузера.
- Отправка запросов к LLM API.
- Запуск watchdog-процесса для восстановления после сбоев.
- Периодическая мягкая перезагрузка для очистки ресурсов.

### 6.2. Фронтенд

Фронтенд реализуется на React и отвечает за пользовательский интерфейс.

Основные обязанности:

- Отображение экранов приложения.
- Визуализация дневной, недельной и годовой активности.
- Работа с настройками пользователя.
- Взаимодействие с бэкендом через API, предоставляемое Tauri.

### 6.3. Расширение браузера

Расширение реализуется на JavaScript.

Основные обязанности:

- Отслеживание URL активной вкладки.
- Отслеживание заголовка активной вкладки.
- Отправка URL и заголовка активной вкладки на локальный HTTP bridge приложения.

### 6.4. База данных

SQLite хранит:

- логи активности;
- логи простоя;
- настройки пользователя;
- стоп-листы приложений и сайтов;
- сохраненные результаты LLM-группировки.

## 7. Функциональные требования

### 7.1. Ядро трекинга

- Использовать системные хуки ОС для определения смены активного окна.
- Фиксировать начало и окончание каждого интервала активности.
- Сохранять название приложения, заголовок окна и URL, если активность относится к браузеру.
- Останавливать трекинг, если нет активности мыши или клавиатуры более 10 минут.

Текущая реализация первого слоя использует Windows foreground-event hook для быстрой реакции на смену активного окна и оставляет polling раз в 2 секунды как резервный слой. Активные интервалы короче 5 секунд не сохраняются, чтобы не засорять базу шумом от случайных переключений окон. Если Windows сообщает отсутствие ввода мыши и клавиатуры дольше 10 минут, текущая активная сессия закрывается временем последнего ввода, начинается idle-интервал, а новые активные интервалы не пишутся до возвращения активности. Первый review-flow простоя позволяет сохранить заметку, игнорировать idle-интервал или отложить уточнение без изменения записи; модалка автопросмотра относится только к текущему дню и не должна перехватывать экран при просмотре прошлых дат. Базовый URL браузера поступает через локальный browser bridge. Трекер также разрывает активную сессию при длинном системном разрыве цикла, чтобы sleep/wake не склеивал сон компьютера с рабочим интервалом. Более полные системные хуки и упаковка расширения остаются следующими шагами ядра трекинга.

При закрытии окна Tauri приложение явно останавливает трекер, чтобы текущая активная или idle-сессия была закрыта до завершения процесса. У `Tracker` также есть `Drop`-страховка, которая пытается остановить фоновый поток при уничтожении состояния приложения.

Приложения из `stoplist` с типом `app` сравниваются с именем процесса без учета регистра. Если активное окно относится к stoplist-приложению, текущая сессия закрывается, а новая не создается до перехода в разрешенное окно. Элементы с типом `site` хранятся в той же таблице и управляются через экран настроек как доменные исключения для browser bridge.

### 7.2. Уточнение простоя

После завершения периода простоя приложение должно показывать модальное окно:

> Я зафиксировал перерыв с [время начала] по [время окончания]. Чем вы занимались?

Пользователь может:

- выбрать один из потоков, например `Работа` или `Обучение`;
- проигнорировать период простоя;
- при необходимости оставить простое описание активности.

### 7.3. Стабильность

- Реализовать watchdog-процесс, который перезапускает основной процесс трекинга при сбое.
- Реализовать мягкую перезагрузку один раз в 24 часа для предотвращения утечек памяти.
- Перезагрузка не должна приводить к потере уже собранных данных.

## 8. База данных

### 8.1. Схема SQLite

```sql
CREATE TABLE activity_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    app_name TEXT NOT NULL,
    window_title TEXT,
    url TEXT
);

CREATE TABLE idle_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    note TEXT,
    ignored INTEGER NOT NULL DEFAULT 0,
    reviewed INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT
);

CREATE TABLE stoplist (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    type TEXT NOT NULL, -- 'app' or 'site'
    value TEXT NOT NULL UNIQUE
);

CREATE TABLE llm_summary (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    date_key TEXT NOT NULL,
    payload_signature TEXT NOT NULL,
    provider TEXT NOT NULL,
    model TEXT NOT NULL,
    groups_json TEXT NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(date_key, payload_signature, provider, model)
);
```

## 9. Пользовательский интерфейс

### 9.1. Главный экран

Компоновка должна быть близка к Dayflow.

Главный экран содержит:

- переключатель дней;
- вертикальный таймлайн дня с разбивкой по часам;
- блок потоков;
- карточки потоков с суммарным временем и списком стримов.

### 9.2. Блок потоков

Блок потоков отображает результат LLM-саммаризации.

Требования:

- Потоки отображаются набором карточек.
- Каждая карточка представляет один поток, например `Работа` или `Обучение`.
- Внутри карточки показывается общее время потока.
- Внутри карточки показывается список стримов, сгруппированных в этот поток.
- Каждый стрим кликабелен.
- При клике на стрим открывается модальное окно со списком конкретных активностей, вошедших в этот стрим.

### 9.3. Экран истории активности

Экран истории активности содержит:

- компоновку в стиле Dayflow;
- годовую сетку активности в стиле GitHub heatmap;
- горизонтальный таймлайн недельной активности.

Правила heatmap:

- каждая ячейка соответствует одному дню;
- цвет ячейки зависит от общего времени активности за день;
- сетка строится в прямом календарном порядке от января к декабрю, по неделям года;
- сверху отображаются месяцы, а справа от heatmap допустима только агрегированная годовая сводка;
- пустые или почти пустые дни должны визуально отличаться от активных дней.

### 9.4. Экран настроек

Экран настроек должен поддерживать:

- выбор языка;
- выбор темы;
- настройку автозапуска;
- управление стоп-листами;
- экспорт данных в JSON;
- настройку LLM API;
- выбор провайдера LLM;
- ввод API-ключа;
- ввод адреса Ollama для локального режима.
- ввод имени локальной модели Ollama.

## 10. Логика LLM-саммаризации

### 10.1. Иерархия

LLM должна группировать активность в два уровня.

Текущая реализация MVP использует локальную Ollama как первый провайдер. Запрос запускается вручную с главного экрана, чтобы не тратить ресурсы модели на каждое обновление логов. Ответ ожидается в формате JSON с группами `stream_name`, `flow_name` и индексами исходных активностей. После успешного ответа результат сохраняется в `llm_summary` по дате, сигнатуре payload, провайдеру и модели; при повторном открытии того же дня FocusFlow сначала использует сохраненную группировку.

**Уровень 1: стрим**

LLM группирует сырые логи активности в конкретные проекты или задачи. Для анализа используются названия приложений, заголовки окон и URL.

Примеры стримов:

- `Разработка трекера FocusFlow`;
- `Анализ конкурентов`;
- `Рабочая коммуникация`.

**Уровень 2: поток**

LLM присваивает каждому стриму высокоуровневую категорию.

Примеры потоков:

- `Работа`;
- `Обучение`;
- `Общение`;
- `Развлечения`;
- `Рутина`.

### 10.2. Шаблон промпта

Использовать следующий шаблон для запроса к LLM API:

```plaintext
Ты - ассистент по продуктивности. Проанализируй лог активности пользователя и сгруппируй задачи в "Стримы" (конкретные проекты), а затем определи для каждого стрима "Поток" (общую категорию: "Работа", "Обучение", "Общение", "Развлечения", "Рутина").

Пример 1:
Вход:
[
  {"app": "Code.exe", "title": "spec.md - FocusFlow", "duration": 25},
  {"app": "chrome.exe", "title": "Tauri documentation", "duration": 10}
]

Выход:
[
  {"stream_name": "Разработка FocusFlow", "flow_name": "Работа", "activities": [0, 1]}
]

Пример 2:
Вход:
[
  {"app": "slack.exe", "title": "general", "duration": 5},
  {"app": "chrome.exe", "title": "Gmail - Входящие", "duration": 15}
]

Выход:
[
  {"stream_name": "Рабочая коммуникация", "flow_name": "Общение", "activities": [0, 1]}
]

Твоя задача:

Вход:
[...список активностей в формате JSON...]

Выход:
Верни только JSON без пояснений.
[
  {
    "stream_name": "Название стрима 1",
    "flow_name": "Название потока 1",
    "activities": [индексы активностей из входного списка]
  },
  {
    "stream_name": "Название стрима 2",
    "flow_name": "Название потока 2",
    "activities": [индексы активностей из входного списка]
  }
]
```
