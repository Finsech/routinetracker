# Спецификация отложенного оркестратора разработки

## 1. Статус

Оркестратор **не реализуется на текущем этапе**.

Текущий фокус проекта - создание MVP FocusFlow без дополнительного слоя автоматизации. Эта спецификация нужна, чтобы сохранить идею и вернуться к ней позже без потери контекста.

## 2. Назначение

Будущий оркестратор должен помогать в разработке FocusFlow, но не заменять инженерный контроль.

Его основная задача - автоматизировать повторяемые операции:

- анализ спецификации проекта;
- выявление недостающих требований;
- подготовка плана работ;
- генерация отчетов;
- запуск разрешенных команд сборки и проверки;
- анализ ошибок сборки;
- подготовка предлагаемых исправлений;
- ведение истории запусков.

## 3. Что оркестратор не должен делать

Оркестратор не должен:

- самовольно перезаписывать существующие файлы;
- выполнять произвольные команды, предложенные LLM;
- отправлять в LLM секреты, `.env`, API-ключи и приватные токены;
- смешивать собственную спецификацию со спецификацией FocusFlow;
- создавать проект с нуля без официального scaffold, если доступен надежный шаблон;
- применять исправления без предварительного отчета и разрешения пользователя.

## 4. Базовый подход

Оркестратор должен быть одноразовым инструментом под FocusFlow.

Архитектурный принцип:

1. Основной проект создается через официальный Tauri + React scaffold.
2. LLM используется для прикладного кода, планирования и анализа ошибок.
3. Все действия проходят через dry-run/report.
4. Запись на диск и запуск команд выполняются только после явного разрешения пользователя.

## 5. Провайдер LLM

Стартовый провайдер - Gemini.

Рекомендуемый подход:

- Gemini 2.5 Pro - для архитектурных решений, генерации кода и исправления ошибок;
- более быстрые модели - только для простых задач после появления стабильного pipeline;
- Ollama - как будущий локальный backend, если потребуется приватность или снижение стоимости.

Причина выбора Gemini для первой версии: оркестратору нужны сильное reasoning-качество, надежная генерация кода и структурированные ответы.

## 6. Роли оркестратора

Роли должны быть реализованы как фазы pipeline, а не как хаотичный набор промптов.

Планируемые роли:

- **Spec Analyst** - читает спецификацию FocusFlow и ищет пробелы.
- **Architect** - предлагает структуру работ и технические решения.
- **Planner** - формирует пошаговый план реализации.
- **Generator** - готовит код или содержимое файлов.
- **Reviewer** - проверяет ответы LLM перед применением.
- **Builder** - запускает разрешенные команды сборки и проверки.
- **Fixer** - предлагает исправления по логам ошибок.
- **Reporter** - собирает итоговый отчет.

## 7. Безопасность

### 7.1. Dry-run по умолчанию

Любое изменение файлов должно сначала попадать в отчет:

- какие файлы будут созданы;
- какие файлы будут изменены;
- какие команды будут запущены;
- почему это изменение нужно;
- какие риски есть.

### 7.2. Запрет на произвольные команды

Команды запускаются только через allowlist.

Предварительный allowlist:

- `npm create tauri-app`;
- `npm install`;
- `npm run build`;
- `npm run tauri build`;
- `cargo check`;
- `cargo fmt`;
- `cargo clippy`;
- `cargo test`;
- `npm run lint`;
- `npm run test`.

Любая команда вне allowlist должна требовать отдельного решения.

### 7.3. Запрет на секреты

Оркестратор не должен отправлять в LLM:

- `.env`;
- API-ключи;
- токены;
- приватные сертификаты;
- содержимое credential-файлов;
- чувствительные локальные данные пользователя.

## 8. История запусков

Каждый запуск должен сохраняться в отдельную папку:

```text
orchestrator/runs/<timestamp>/
```

Внутри запуска нужно хранить:

- входную спецификацию;
- найденные вопросы;
- generation plan;
- промпты;
- ответы модели;
- dry-run отчет;
- список разрешенных действий;
- логи команд;
- итоговый отчет;
- список созданных и измененных файлов.

## 9. Рекомендуемая структура будущего оркестратора

```text
orchestrator/
  README.md
  ORCHESTRATOR_SPEC.md
  .env.example
  requirements.txt
  config.yaml
  orchestrator.py
  prompts/
    spec_analyst.md
    architect.md
    planner.md
    generator.md
    reviewer.md
    fixer.md
    reporter.md
  runs/
```

На текущем этапе эта структура не создается полностью, чтобы не превращать идею в незавершенный второй проект.

## 10. Условия возврата к реализации

К реализации оркестратора стоит вернуться, когда выполнены минимум три условия:

- у FocusFlow есть рабочий MVP;
- есть стабильные команды сборки и проверки;
- появились повторяемые задачи, которые выгодно автоматизировать;
- есть базовые тесты или хотя бы надежный build pipeline;
- понятно, какие части проекта можно менять автоматически.

## 11. Первое возможное MVP оркестратора

Минимальная версия оркестратора должна уметь:

1. Читать `SPECIFICATION.md`.
2. Анализировать пробелы в требованиях.
3. Задавать уточняющие вопросы.
4. Формировать план работ.
5. Записывать отчет в `orchestrator/runs/<timestamp>/report.md`.

На первом MVP оркестратор не должен писать код продукта и не должен применять исправления автоматически.
